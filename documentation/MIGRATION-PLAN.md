# План миграции: движок и контент MedControl

Краткая выжимка из переписки по разделению репозиториев. Подробности и обоснования — в [CHAT-TRANSCRIPT.md](./CHAT-TRANSCRIPT.md).

## В чём главная проблема сейчас

Один репозиторий смешивает приложение (React, роутинг, поиск) и материалы MedControl (статьи, картинки, новости). Часть «описания сайта» (меню, карточки главной, превью) лежит в коде движка, хотя по смыслу это конфиг продукта.

**Инвариант текущего движка:** адрес страницы совпадает с путём к файлу `.md` под `public/content/` (без расширения). Любой перенос или переименование `.md` без синхронного обновления навигации и ссылок ломает сайт.

---

## Три зоны ответственности

### A. Уже сейчас без доработки движка

Допустимо только то, что **не меняет** путь к `.md` относительно `public/content/`.

| Действие | Риск |
|----------|------|
| Базовая линия: вывод `check-internal-links.mjs`, список `path` из `flatArticles` | Нет |
| Правки текста в статьях без смены имён файлов и путей к картинкам | Низкий |
| Переименование картинок + обновление ссылок в `.md` (пути остаются под `/content/images/...`) | Средний, если пропустить ссылку |
| README и внутренняя документация в контент-репо | Нет |

**Нельзя** без согласованных правок движка и навигации: перенос `.md`, переименование `.md`, новые корневые имена вместо `obshee` / `admin` / и т.д.

### B. Потребует изменений в коде движка

- Разделение на два Git-репозитория + submodule или копирование контента в CI перед сборкой.
- Любая смена slug (пути статьи): `nav.js`, ссылки в `.md`, поиск, PDF.
- Вынос `nav.js` в JSON/YAML в контент-репо.
- Вынос дашборда и превью карточек в конфиг + медиа в контент.
- Нейтрализация бренда в `index.html`, темах, стилях.

### C. Классификация

| Тип | Примеры |
|-----|---------|
| **Движок** | Роутинг, layout, рендер Markdown, поиск (логика), PDF, UI без текстов продукта |
| **Контент** | `.md`, `images/`, `News/` |
| **Контентный конфиг** | `nav.js`, дашборд, превью разделов, `searchMetadata`, `news-tree.json`, заголовок вкладки, иконки дашборда |

## Пошаговый план миграции

**Шаг 0 — базовая линия (монорепо движка)**  
Запустить `node docs-app/scripts/check-internal-links.mjs`, сохранить вывод; выпиской зафиксировать все `path` из `flatArticles` и сверить с `public/content/<path>.md`. Артефакт: `baseline-links-before-migration.txt` (живёт в репозитории движка на момент съёмки).

**Шаг 1 — контент-репозиторий (этот репо)**  
Содержимое `docs-app/public/content/` в **корень** контент-репо **без** лишней вложенности `content/content/…`. Первый коммит — копия 1:1.

**Шаг 2 — подключить контент к движку** _(рискованно)_  
Submodule в `docs-app/public/content` или CI: checkout контента + копирование перед `npm ci` / `build`.

**Шаг 3 — вынести навигацию в конфиг контента** _(рискованно)_  
Например `content/config/nav.json`; движок подхватывает при сборке или через `public/`.

**Шаг 4 — дашборд и превью**  
JSON-конфиги + картинки в контент-репо.

**Шаг 5 — бренд и тема**  
Вынести из обязательных мест в движке.

**Шаг 6 — четыре корня и новая иерархия** _(рискованно)_  
Только после согласованного обновления навигации и массовой правки ссылок. Отдельно решить судьбу `glossariy` и `News`.

## Целевая структура контент-репозитория (после полной миграции)

```text
корень репозитория
  README.md
  documentation/     # план миграции, архив чата, REPO-STRUCTURE
  docs/                # контент сайта: glossary, obshee, admin, medkabinet, medadmin
  news/
  references/        # не страницы сайта
  config/              # опционально после выноса nav и т.д.
```

На этапе первого импорта достаточно текущего дерева без `config/`.

## Целевая структура репозитория движка

```text
docs-app/
  public/
    content/    # .gitkeep или submodule → medcontrol-docs-content
  src/
  ...
README.md       # как подтянуть контент и собрать
.github/workflows/...
```

## Риски (сводка)

- **Высокие:** разрыв CI и контента; смена путей `.md` без обновления навигации и ссылок; кривой импорт `nav` из JSON в Vite.
- **Средние:** перенос картинок дашборда; регрессия вёрстки при обезличивании CSS.
- **Низкие:** смена `<title>` через переменные окружения.

## Чек-лист миграции

- [ ] Сохранён вывод проверки ссылок «до».
- [ ] Создан репозиторий контента; перенесено дерево `public/content` 1:1.
- [ ] Движок собирается с подставленным контентом (submodule или CI).
- [ ] Обновлён workflow деплоя; пробный деплой.
- [ ] Навигация вынесена в контент-конфиг (когда сделает разработка).
- [ ] Дашборд и превью вынесены.
- [ ] MedControl убран из обязательных мест в движке.
- [ ] (Опционально позже) физическое переименование корней + обновление конфига и ссылок.
- [ ] README в обоих репозиториях.

## Чек-лист проверки после изменений

- [ ] `npm ci` и `npm run build` в движке с актуальным контентом.
- [ ] `node docs-app/scripts/check-internal-links.mjs` без ошибок.
- [ ] Главная, карточки, статьи по разделам, новости.
- [ ] Поиск, PDF (если используются).
- [ ] Чистый клон движка по README + подключение контента воспроизводимо.

## Что можно удалить в репозитории движка после стыковки

Когда контент подключается только из `medcontrol-docs-content` (submodule или CI):

- Удалить **отслеживаемые** файлы внутри `docs-app/public/content/`, оставив placeholder (например `.gitkeep`) или сам submodule.
- Не дублировать в движке: полное дерево статей, `images/`, `News/` — источник правды здесь, в `medcontrol-docs-content`.

Документацию о разделении и переписку держите в **этом** репозитории (`docs/`), чтобы в движке не хранить дубликаты объяснений.
